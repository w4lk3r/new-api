# 邀请奖励机制技术设计

## 1. 文档目的

本文档记录系统邀请奖励机制的业务规则、现有实现以及后续扩展方案，作为后端、前端和运营配置的实现依据。

邀请奖励包含两种方式：

1. 被邀请用户注册成功后，邀请者获得一次性佣金奖励。
2. 被邀请用户后续充值成功后，邀请者按照充值金额的一定比例获得返佣。

两种奖励可以独立启用，也可以同时启用。

## 2. 现有数据和功能

用户模型中已经包含以下字段：

| 字段 | 含义 |
| --- | --- |
| `aff_code` | 用户自己的邀请/推荐码，具有唯一索引 |
| `inviter_id` | 注册时使用邀请链接的用户 ID |
| `aff_count` | 成功邀请人数 |
| `aff_quota` | 当前可提取的邀请奖励额度 |
| `aff_history_quota` | 历史累计获得的邀请奖励额度 |
| `quota` | 用户主账户余额 |

用户创建时会自动生成随机邀请码。新用户通过邀请链接注册后，系统根据邀请码解析邀请者，并保存被邀请用户的 `inviter_id`。

当前注册奖励由以下配置控制：

- `QuotaForInvitee`：被邀请者注册后获得的额度。
- `QuotaForInviter`：邀请者因成功邀请获得的额度。

将 `QuotaForInvitee` 设置为 `0`，即可关闭被邀请者的注册赠送。将 `QuotaForInviter` 设置为 `0`，即可关闭邀请者的注册一次性奖励。

当前用户可以在钱包中查看待领取奖励、历史累计奖励和邀请人数，并通过“转入余额”将部分 `aff_quota` 转入自己的 `quota`。钱包中的“查看详情”可以查看被邀请用户和奖励记录，但采用隐私脱敏视图：用户名脱敏，只显示日期，不显示充值原始额度、返佣比例和精确时间。转入操作使用事务和行锁，避免并发操作造成额度重复使用。

## 3. 奖励账户规则

邀请奖励与主余额分开存储：

```text
aff_quota          当前可领取/可提取的邀请奖励
aff_history_quota  历史累计获得的邀请奖励
quota              用户主余额
```

发放奖励时：

```text
aff_quota += reward
aff_history_quota += reward
```

用户自主转入主余额时：

```text
aff_quota -= transfer
quota += transfer
```

管理员线下向用户付款后执行“提取清零”时，建议只执行：

```text
aff_quota = 0
```

同时保留 `aff_history_quota`，并写入管理员操作日志。管理员提取清零不应自动增加 `quota`，避免与线下付款重复入账。

## 4. 奖励方式一：注册一次性佣金

### 4.1 业务规则

当用户通过有效邀请链接注册成功，并且系统允许邀请奖励时：

1. 记录被邀请用户的 `inviter_id`。
2. 按 `QuotaForInvitee` 向被邀请者发放注册赠送额度。
3. 按 `QuotaForInviter` 向邀请者发放一次性佣金。
4. 邀请者的 `aff_count` 增加 1。

当前实现将“建立有效邀请关系”和“发放一次性奖励”分开处理：即使 `QuotaForInviter = 0`，仍会记录邀请流水并增加 `aff_count`；只有满足奖励条件时才增加 `aff_quota` 和 `aff_history_quota`。这样可以关闭注册赠送，同时保留真实邀请人数统计。

### 4.2 配置示例

```text
QuotaForInvitee = 0       # 被邀请者注册不赠送
QuotaForInviter = 100000  # 邀请者获得固定额度，具体单位遵循系统 quota 规则
```

如果两项都为 `0`，则只建立邀请关系，不发放注册奖励，但仍会统计有效邀请人数。

## 5. 奖励方式二：充值按比例返佣

### 5.1 业务规则

被邀请用户充值成功后，邀请者按照配置比例获得返佣。例如：

```text
被邀请用户实际到账充值额度 = 100
返佣比例 = 10%
邀请者获得奖励 = 10
```

奖励不直接进入邀请者主余额，而是先累积到邀请者的 `aff_quota` 和 `aff_history_quota`，由用户或管理员后续提取。

### 5.2 推荐配置

充值返佣使用以下独立配置：

```text
payment_setting.invite_recharge_reward_enabled = false
payment_setting.invite_recharge_reward_rate    = 0
```

后续如需按用户组或渠道配置，再将比例扩展到用户组、充值方式或活动规则，而不是一开始复制多套结算逻辑。

### 5.3 计奖基数

默认建议按“充值成功后实际到账的 quota”作为计奖基数：

```text
reward = credited_quota * reward_rate
```

这样可以自然兼容折扣、赠送、不同货币显示方式和系统内部额度换算。若业务要求按实际支付金额计奖，则应先统一转换为系统 quota 单位，并明确汇率、折扣和手续费的处理方式。

以下额度通常不应触发充值返佣：

- 管理员手动增加的额度
- 新用户注册赠送额度
- 邀请注册赠送额度
- 兑换码或活动赠送额度（除非业务明确允许）
- 退款、冲正或失败订单

### 5.4 充值成功后的处理顺序

建议在充值订单确认成功、用户主余额入账成功之后执行返佣：

1. 确认订单处于最终成功状态。
2. 确认该订单尚未发放过邀请返佣。
3. 查询充值用户的 `inviter_id`。
4. 校验邀请者仍然存在且账户状态允许领取奖励。
5. 根据充值基数和返佣比例计算奖励。
6. 使用事务和行锁更新邀请者的 `aff_quota` 与 `aff_history_quota`。
7. 写入返佣明细和操作日志。

返佣失败不应回滚已经成功的用户充值，但必须记录可重试的结算状态，避免出现用户已充值而返佣静默丢失的情况。

## 6. 奖励流水数据模型和展示边界

系统使用 `invitation_reward_records` 保存注册奖励和充值返佣流水，用于追溯和幂等：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `inviter_id` | 获得奖励的邀请者 ID |
| `invitee_id` | 产生奖励的被邀请者 ID |
| `type` | `signup` 或 `recharge` |
| `source_id` | 注册事件 ID、充值订单 ID 等来源标识 |
| `base_quota` | 返佣计算基数，注册奖励可为 0 |
| `rate` | 返佣比例，注册奖励可为空或 0 |
| `reward_quota` | 本次奖励额度 |
| `status` | 当前使用 `settled`、`failed`，退款功能上线后可扩展 `reversed` |
| `created_at` | 创建时间 |
| `settled_at` | 结算时间 |
| `remark` | 运营或异常说明 |

建议对 `type + source_id` 建立唯一约束，确保同一注册事件或充值订单不会重复发奖。数据库实现必须同时兼容 SQLite、MySQL 和 PostgreSQL。

### 6.1 内部明细与用户展示分离

系统保留每笔奖励的完整内部流水，用于管理员审计、幂等判断和后续退款冲正；普通邀请者不能直接读取这张完整流水表。

用户端通过单独的脱敏查询接口获取展示数据：

| 用户端字段 | 展示规则 |
| --- | --- |
| 被邀请用户 | 用户名脱敏，例如 `private-user` 展示为 `p***r` |
| 注册/奖励时间 | 只显示 `YYYY-MM-DD` 日期 |
| 充值原始额度 | 不返回、不展示 |
| 返佣比例 | 不返回、不展示 |
| 本次奖励额度 | 可以展示 |
| 奖励类型 | 展示为注册奖励或充值返佣 |

因此，当前不需要按月聚合或新增月度汇总表：数据库继续按笔保存完整流水，用户端只读取隐私安全的投影视图。后续如果采用月结算，再在现有明细之上增加月度汇总或结算表。

当前接口分为两组：

- 普通用户：`GET /api/user/invited-users`、`GET /api/user/invitation-rewards`，只返回脱敏投影视图。
- 管理员：`GET /api/user/:id/invited-users`、`GET /api/user/:id/invitation-rewards`，返回完整管理明细。

## 7. 退款和冲正

如果充值订单退款，建议按原返佣流水进行冲正，而不是重新计算当前比例：

1. 找到原充值返佣记录。
2. 将原记录标记为 `reversed`。
3. 从邀请者的可用奖励中扣除相应返佣。
4. 如果奖励已经被转入主余额或被管理员提取，应进入负债/人工审核流程，不能直接让余额产生负数。

因此，正式实现前需要明确：已提取的返佣是否允许从用户主余额扣回，还是由管理员人工处理。

## 8. 安全、合规和结算约束

- 只有最终成功的充值订单才能触发返佣。
- 充值回调必须幂等，不能按重复回调次数重复发奖。
- 返佣比例必须限制在合法范围内，不能允许负数、无穷大或异常值。
- 返佣计算应使用统一的 quota 换算工具，避免浮点数转换和整数溢出产生异常额度。
- 应记录奖励来源、计算基数、比例、奖励额度和结算状态。
- 管理员提取清零必须校验管理员权限，并记录操作人、操作前额度和操作后额度。
- 支付、邀请奖励和返佣功能应继续受系统合规确认开关控制。
- 应设置单用户、单邀请者或周期级奖励上限，降低刷号和自充套利风险。

## 9. 前端功能规划

### 用户端

现有推荐奖励卡片可以继续展示：

- 待领取奖励
- 历史累计奖励
- 邀请人数
- 推荐链接
- 转入余额按钮
- “查看详情”入口
- 被邀请用户脱敏列表
- 注册奖励和充值返佣记录
- 仅显示日期和奖励额度的隐私安全视图

用户端不显示被邀请者的充值原始金额、返佣比例和精确充值时间，避免泄露被邀请者的消费隐私。充值返佣仍然可以按笔展示奖励结果；如果未来改为月度结算，只需替换用户端查询层，内部逐笔流水继续保留。

### 管理员端

建议增加：

- 按邀请者查询被邀请用户列表
- 查看每个被邀请用户的注册时间、充值金额和产生的返佣（管理员权限范围内）
- 查看邀请奖励流水
- “提取清零”按钮
- 提取清零二次确认和操作日志
- 返佣比例、开关和奖励上限配置

## 10. 当前实现状态

已经实现：

1. 注册一次性奖励及对应奖励流水。
2. 充值成功后按实际到账 quota 比例返佣。
3. 基于奖励类型和来源 ID 的唯一幂等约束。
4. Stripe、Creem、易支付、Waffo、Waffo Pancake 和管理员补单的返佣接入。
5. 管理员查看受邀用户、查看奖励流水以及“标记已支付并清零”。
6. 全局充值返佣开关和返佣比例配置。
7. 普通用户在钱包查看脱敏后的邀请用户列表和奖励记录。
8. 用户端与管理员端展示分离：管理员可查看完整充值基数、返佣比例和精确时间，普通用户只能查看脱敏后的日期和奖励额度。

尚未实现、需要在退款业务明确后继续开发：

1. 充值退款后的自动返佣冲正。
2. 已提取返佣的负债或人工审核流程。
3. 用户组、充值方式和活动级差异化返佣比例。
4. 用户端按月汇总结算、充值退款后的自动返佣冲正。
